# Bundle Format and Signed-Data Migration Policy

RyeOS accepts one bundle format at a time. The current formats are:

- signed bundle manifests: `ryeos.bundle-manifest/v1`;
- signed node bundle registrations: `ryeos.bundle-registration/v1`.

Manifest v1 predates an on-wire format field. Its discriminator is therefore
its closed YAML schema: a mapping with required `name`, `version`,
`provides_kinds`, and `requires_kinds` fields, the currently defined optional
fields, and no unknown fields. Registration v1 is a closed YAML mapping with a
required `kind: node`, an absolute `path`, and optional
`command_registration_caps`. Both records must use the current RyeOS YAML
signature envelope and pass trusted-signature verification at installed-data
boundaries.

There is no legacy reader, field alias, best-effort coercion, or automatic
rewrite of signed data. Missing current required fields, unknown fields, and
future or old shapes fail closed. `manifest.source.yaml` is unsigned publisher
input and may be materialized only by the existing development/publish path;
it is not accepted as an installed signed manifest.

## Changing a format

A format change is a coordinated release operation:

1. Define one replacement format and parser; do not broaden the current parser.
2. Regenerate every affected manifest or registration from authoritative
   source data.
3. Sign the exact regenerated bytes with the appropriate publisher or node
   identity.
4. Verify the complete release bundle set with the new reader before shipping.
5. Switch the accepted-format constant and reader in the same release.

RyeOS does not mutate a signed record in place because that would invalidate
its signature and obscure provenance. Operators must republish or rerun the
authoritative registration operation when a format transition is required.
