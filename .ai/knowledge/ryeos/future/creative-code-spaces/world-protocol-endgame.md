# World Protocol Endgame for Signed Creative Spaces

## Purpose

This note captures the far edge of the signed creative spaces idea.

The logical endpoint is metaverse-shaped, but not in the usual sense of one corporate 3D world, VR avatars, land speculation, or virtual malls. The stronger direction is a protocol for making many signed worlds.

```text
not one metaverse
many signed worlds
```

Each world is a bounded, signed, executable context where humans and directives/agents create, inspect, remix, validate, and sync modules.

## Core thesis

Rye should become a substrate for world documents: portable, signed graphs of modules, contracts, events, agents, projections, and provenance.

One sentence:

> Rye lets people create shared spaces out of signed AI-generated modules.

Longer version:

> Rye is a world protocol for AI-generated spaces: prompts become structured modules, modules are signed and validated, nodes attest what belongs in a shared space, peers sync the accepted graph, and the cockpit renders/inspects the result.

## Not one metaverse

The failed metaverse pattern says:

```text
There is one big world. Come put an avatar in it.
```

The Rye pattern should say:

```text
There are many worlds. Each world has its own frame, rules, node policy, visual projections, agents, and accepted modules. Enter one, fork one, remix one, or create a new one.
```

This is closer to:

```text
GitHub + Roblox + Minecraft + package registry + browser + AI coding loop + multiplayer runtime
```

than to a conventional metaverse.

## World documents

The atomic unit may not be an app, game, page, or file. It may be a world document.

A world document is like:

```text
Git repo
+ game save
+ package
+ website
+ social space
+ simulation state
+ agent workspace
```

all as one signed graph.

A world document contains:

- a frame contract;
- accepted modules;
- scene/state graph;
- visual projections;
- directive/entity definitions;
- event history;
- node attestations;
- permissions and policies;
- remix/provenance lineage;
- sync metadata.

You can open it in the cockpit, run it locally, submit changes, fork it, host it on a node, invite others, and ask directives inside it to change it.

## Apps become spaces

Current software often feels like:

```text
open app
use interface
save file
```

This direction shifts the pattern toward:

```text
enter space
talk to entities
modify world
publish signed changes
sync with others
```

A space can be:

- a game;
- an art world;
- a simulation;
- a dashboard;
- a classroom;
- a dev environment;
- a social room;
- a procedural story;
- a research lab;
- a personal operating environment.

The renderer may be Three.js, TUI, graph, timeline, dashboard, document, IDE, or VR later. The world is not the rendering. The world is the signed state plus frame runtime.

## Why AI makes it newly possible

Without AI, making worlds is too expensive. It requires programmers, modelers, designers, writers, server tooling, moderation tooling, and deployment infrastructure.

AI collapses authoring cost.

A person can ask for weird software directly:

```text
Make a floating island with a lighthouse that flickers when someone nearby has an unresolved task.
```

The AI can produce structured module data. Rye can validate, sign, store, attest, sync, and render it.

The important pipeline is:

```text
prompted intent
  -> structured module
  -> validation
  -> signed existence
  -> shared projection
```

This is validated imagination: imagination converted into structured, signed, operational world state.

## Beyond user-generated content

Traditional UGC is often:

```text
user creates asset
platform stores asset
other users consume asset
```

Signed creative spaces should be active UGC:

```text
user creates module
module has behavior/projection/provenance
module can be validated
module can be remixed
module can contain agents
module can interact with other modules
module can move across nodes
```

The content is not just media. It is live structured state.

## Frames are constitutions

A frame is not just a schema. It is the constitution of a world.

It defines:

- what can exist;
- what can happen;
- who can publish;
- what needs approval;
- how time works;
- what directives can do;
- how visuals are interpreted;
- what safety limits apply;
- what counts as accepted reality.

A hard-scifi frame has one constitution. A surreal art frame has another. A classroom frame, codebase frame, dungeon frame, and research frame each have their own.

This gives communities cultural control without forcing one universal world policy.

## Nodes are city-states

A node is not just infrastructure. It is a city-state, gallery, journal, server, lab, classroom, or studio.

A node says:

```text
These modules are accepted here.
```

It has:

- admission policy;
- curators;
- trusted signers;
- moderation rules;
- resource budgets;
- featured works;
- sync peers;
- attestation keys.

People choose nodes based on taste, trust, strictness, culture, safety, and community.

Possible node types:

- official seed nodes;
- friend nodes;
- artist collectives;
- hard simulation nodes;
- experimental chaos nodes;
- educational nodes;
- private studio nodes;
- research nodes.

The platform does not need one universal authority. It needs portable identity, verifiable acceptance, and configurable trust.

## Identity becomes creative gravity

A signing key is not just login. It is authorship.

Over time, a creator key accumulates:

- objects created;
- styles published;
- modules accepted;
- worlds founded;
- directives authored;
- remixes made;
- nodes trusted;
- collaborations joined.

This becomes reputation through provenance, not necessarily tokens or financialization.

You see a module and know:

```text
This was made by that key. Their objects are beautiful, weird, stable, dangerous, useful, rigorous, or influential.
```

## Directives are inhabitants

Directives are the differentiator from ordinary world-building platforms.

In Rye, inhabitants can be:

- humans;
- directives;
- graphs;
- schedulers;
- simulated societies;
- curator agents;
- worldsmith agents;
- debugger agents;
- tutor agents;
- research agents;
- NPC-like entities backed by model context.

A directive is an entity with:

- context;
- intent;
- permissions;
- tools;
- memory/state;
- model choice;
- budget;
- scope.

Examples:

### Art world

- curator explains pieces;
- lighting agent adjusts mood;
- remix agent helps make variants.

### Cosmos world

- ecologist evolves biomes;
- orbital agent warns about unstable systems;
- civilization agent speaks as a society.

### Codebase world

- test agent lives near failing modules;
- refactor agent proposes paths;
- release agent guards deployment gates.

### Classroom world

- tutor agent adapts lessons;
- examiner validates submissions;
- historian explains prior student worlds.

A directive is not just an NPC. It is scoped operational intelligence with permissions.

## Time gives worlds memory

Static scenes are useful, but time makes worlds matter.

Time should be represented through signed event chains and replayable graphs, not uncontrolled mutable state.

Worlds change through:

```text
draft event
  -> validated event
  -> signed event
  -> attested event
  -> synced event
  -> rendered event
```

This means world history is inspectable.

More strongly: time can be graph replay.

A graph is a temporal structure:

```text
initial state
  -> step A
  -> branch B1 / B2
  -> directive decision
  -> validation gate
  -> signed event
  -> attested state
```

If every graph node records its inputs, outputs, module hashes, directive version, frame version, seed, validation result, signer, and node attestation, then the world can replay its own history.

Replay can be used to:

- inspect why something happened;
- reconstruct an older state;
- fork the world from a past graph node;
- compare alternate branches;
- verify a node's attested outcome;
- debug a directive/entity decision;
- turn history into a visible timeline in the cockpit.

In this model, time is not just a clock. Time is the traversable execution graph of signed changes.

A user can ask:

```text
Why did this moon turn blue?
```

The system can answer:

```text
tick 182: artist module applied blue-glass material
tick 203: ecologist directive added fungal glow
tick 211: node accepted remix by Mira
```

Worlds have memory because every accepted change has provenance.

## Cockpit as world devtools

The cockpit becomes the browser/editor/devtools for signed worlds.

It can show:

- visual scene;
- module graph;
- provenance chain;
- node validation status;
- directive conversations;
- time/event history;
- TUI projection;
- source module data;
- remix tree;
- sync status.

This is not only a game UI. It is more like:

```text
Chrome DevTools for living worlds
GitHub + Unity editor + terminal + multiplayer client
```

The web cockpit provides the visual/spatial magic. The TUI cockpit provides operator/hacker/provenance depth.

## Interoperability should be substrate-level

Naive metaverse interoperability says:

```text
A sword from one world should work in every other world.
```

That is not realistic.

Useful interoperability is lower-level:

- same identity model;
- same signature verification;
- same CAS object model;
- same attestation pattern;
- same provenance graph;
- same module packaging;
- same cockpit inspection model;
- same sync protocol.

A sword from a fantasy frame may not be valid in a cosmos frame. But it can still be inspected, credited, forked, translated, referenced, displayed as an artifact, or converted by an agent.

## Translation between worlds

If frames differ, AI can translate intent, style, authorship, and lineage between frames.

Example:

```text
Bring this crystal moon style into my dungeon frame.
```

The translation agent maps:

```text
crystal moon material
  -> dungeon wall material
  -> glowing gem cave
  -> refractive torchlight
```

The original module may not import literally, but its intent and provenance can carry across.

Cross-world magic may come less from universal object portability and more from portability of style, intent, authorship, and lineage.

## Economy should start as reputation and reuse

Avoid making the early story about tokens, land, or speculation.

The natural first economy is:

- attention;
- remix count;
- node acceptance;
- trusted signers;
- featured modules;
- style reuse;
- directive reuse;
- frame adoption;
- curation reputation.

Later, paid nodes, commissioned modules, hosted worlds, premium directives, or professional frames could exist. But the first economy should be creative reputation through provenance.

## Endgame stack

The full stack can be named as layers:

```text
Identity layer:
  keys, signatures, trust

Object layer:
  CAS, modules, manifests, provenance

Contract layer:
  frame schemas, validators, policies

Runtime layer:
  renderers, behavior primitives, simulation engines

Node layer:
  attestation, moderation, indexing, sync

Agent layer:
  directives, graphs, schedulers, scoped entities

Interface layer:
  Three.js, TUI, graph, timeline, editor, API

Social layer:
  remix, collaboration, curation, reputation
```

At the far edge:

- every app is a frame;
- every file is a module;
- every assistant is a directive entity;
- every server is a node;
- every save is a signed event;
- every plugin is an attested module;
- every community is a node policy;
- every UI is a projection.

That is the logical end: not a metaverse as a place, but a metaverse as a software grammar.

## Product discipline

The endgame is enormous, but the wedge must stay tiny.

Do not start by pitching:

```text
peer-to-peer AI-native signed executable metaverse world protocol
```

Start by showing:

```text
I typed "make a moon with blue glass forests."
It appeared in a Three.js scene.
I signed it.
The node accepted it.
You pulled it.
Now you are remixing it.
```

Once people feel that, the deeper substrate becomes believable.
