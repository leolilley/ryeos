# World Protocol Endgame for RyeOS Digital Spaces

## Purpose

This note captures the far edge of the RyeOS digital spaces direction.

The endpoint is metaverse-shaped only if “metaverse” means many independently owned, signed, portable worlds. It is not one corporate 3D world, one account system, one marketplace, or one global server.

```text
not one metaverse
many signed worlds
```

## Core thesis

A world is not a server-owned database. It is a signed CAS graph interpreted by a frame.

A hosted node can keep that world online, admit changes, publish heads, serve subscriptions, and run jobs. But durable reality remains verifiable signed state.

The Cockpit is the home dimension where a principal sees, opens, inspects, remixes, hosts, and syncs those worlds.

## Worlds as digital places with gravity

Current digital spaces often lack weight:

- ownership is a platform row;
- history can be silently rewritten;
- presence is simulated by centralized session state;
- portability depends on export features;
- provenance is hidden or absent.

RyeOS worlds should feel different because the underlying facts are signed and content-addressed.

The geometry can vary, but the state has gravity:

- things are where their hashes say they are;
- authorship is signed;
- history is replayable;
- policies are inspectable;
- hosted nodes are replaceable;
- renderers are interpretations, not truth.

## World documents

The atomic unit may not be an app, game, page, or file. It may be a world document: a signed object graph that can be entered through portals and rendered through dimensions.

A world document can contain:

- world policy;
- frame contract;
- accepted heads;
- scene/state objects;
- event history;
- dimensions/renderers;
- directive/tool inhabitants;
- hosted node descriptors;
- admission records;
- permissions and grants;
- remix/provenance lineage;
- sync metadata.

It is closer to:

```text
Git repo
+ game save
+ package
+ website
+ social space
+ simulation state
+ agent workspace
```

than to a traditional file.

## Portals, not apps

The user-facing pattern changes from:

```text
open app
use interface
save file
```

to:

```text
open Cockpit
enter portal
inhabit dimension
modify signed state
publish or keep changes
sync with peers or hosted nodes
```

A portal can lead to:

- a game;
- an art world;
- a simulation;
- a codebase world;
- a classroom;
- a music space;
- a memory landscape;
- a collaborative fiction room;
- a research lab;
- a personal garden.

The portal is not the authority. It is an entry point with provenance, policy, and required object closures.

## Dimensions, not one renderer

The same world can be interpreted many ways.

```text
same signed state
  -> Three.js spatial dimension
  -> TUI operator dimension
  -> graph/provenance dimension
  -> timeline/history dimension
  -> editor/devtools dimension
  -> simulation dimension
  -> share-card/web dimension
```

This is the key difference from ordinary games and platforms. The shared truth is semantic and signed; the geometry is a choice.

## Frames as constitutions

A frame is the constitution of a world family.

It defines:

- what can exist;
- what can happen;
- how objects are validated;
- who can publish;
- what needs approval;
- how time works;
- what directives/tools can do;
- how renderers interpret state;
- what safety limits apply;
- what counts as accepted reality.

A hard-scifi simulation frame, garden frame, codebase frame, classroom frame, art frame, and dungeon frame should not share one universal policy. Each frame carries its own rules.

## Hosted nodes as presence, not authority

A hosted node is a high-uptime RyeOS node.

It can act as:

- gallery;
- studio;
- lab;
- classroom;
- game server;
- portal host;
- world mirror;
- subscription hub;
- job runner;
- indexer/search node.

It says:

```text
I host this portal.
I serve these object closures.
I accept these heads under this policy.
I run these jobs under these grants.
```

It does not say:

```text
I am global truth.
I own your identity.
I authorize all RyeOS execution.
```

Nodes can develop culture and reputation through the policies they enforce and objects they admit. People can choose nodes based on trust, taste, reliability, strictness, safety, or community.

## Identity as authorship

A RyeOS principal is not just login. It is authorship and authority.

Over time, a key accumulates:

- worlds founded;
- objects created;
- frames authored;
- renderers built;
- portals hosted;
- policies signed;
- jobs run;
- remixes made;
- nodes trusted;
- collaborations joined.

This becomes reputation through provenance, not a platform-owned score.

App-local realm auth can still exist for visitors. A portal may allow someone to enter with a normal browser session. But signed world authorship and RyeOS protocol authority belong to RyeOS principals.

## Directives and tools as inhabitants

Directives, tools, graphs, and schedulers can become inhabitants of worlds.

Examples:

- `worldsmith` proposes valid world objects;
- `artist` generates styles and material variants;
- `curator` reviews admissions and explains policy;
- `scheduler` advances frame time;
- `ecologist` evolves a garden or biome;
- `test-agent` lives near failing code paths;
- `release-agent` guards a deployment portal;
- `tutor` inhabits a classroom world.

They act through scoped context, permissions, budgets, and signed outputs. They should not receive the whole universe by default.

## Time as replayable graph

World time should be replayable and inspectable.

Instead of uncontrolled mutation, worlds should accumulate signed events, snapshots, admissions, and head updates.

```text
prior head
  -> signed event/object proposal
  -> validation
  -> admission/policy decision
  -> new accepted head
```

This makes history load-bearing. The Cockpit can replay, fork, compare, and explain how a world came to be.

## Economies without platform ownership

If worlds support exchange, ownership, editions, or rewards, those should emerge from signed provenance and policy, not a platform database.

Possible future objects:

- signed edition records;
- ownership/transfer attestations;
- paid hosted presence leases;
- creator support records;
- world object licenses;
- admission fees or contribution policies;
- hosted node service agreements.

Avoid building the financial layer first. The first value is presence, creation, provenance, and portability.

## The endgame

At the far edge:

- every project can become a world;
- every world can have portals;
- every portal can have dimensions;
- every dimension is a projection over signed state;
- every hosted node is a reachable peer, not an authority;
- every meaningful change has provenance;
- every user can return to the Cockpit and inspect what happened.

The pitch is not “decentralized game platform.”

The deeper pitch is:

> digital spaces whose reality is signed, portable, inspectable, and yours.
