# Cluster Bootstrap And Fleet Enrollment

> Trigger: manual identity exchange (`ryeosd auth authorize --identity <file>`) becomes operationally painful at scale — more than a handful of nodes, ephemeral GPU workers, or autoscaling pools.

## Current State

The v3 daemon bootstrap uses explicit file-based trust exchange between nodes. An operator exports a node's public identity document, copies it to a peer, and runs `ryeosd auth authorize` and `ryeosd trust add` on the peer. Reciprocal trust is manual and auditable.

This works for small, stable deployments. It does not scale to:

- fleets of ephemeral GPU workers that spin up and down;
- autoscaling pools where nodes appear without operator intervention;
- environments where pre-distributing identity documents is impractical;
- key rotation across many nodes simultaneously.

## Design: Registry-Backed Enrollment

### Enrollment Authority

Introduce a designated enrollment authority. This is a registry node or a dedicated enrollment service that acts as the root of trust for a fleet.

The enrollment authority:

- holds a long-lived signing key that all fleet nodes trust;
- issues signed node attestations on enrollment;
- maintains an enrollment ledger of all active fleet members;
- publishes revocation lists.

### Join Flow

A new node joins the fleet using a join token.

```
ryeosd join --registry https://registry.example.com --token-file /run/secrets/join-token
```

Join flow:

1. new node generates its own signing key and identity document during `init`;
2. new node presents the join token and its public identity to the enrollment authority;
3. enrollment authority validates the join token;
4. enrollment authority issues a signed attestation for the new node;
5. new node persists the attestation;
6. enrollment authority records the new node in its enrollment ledger;
7. other fleet nodes can verify the new node's attestation against the enrollment authority's public key.

### Join Tokens

Join tokens are short-lived, scoped, and single-use or limited-use.

```json
{
  "token_id": "jt-abc123",
  "issued_by": "fp:registry-fingerprint",
  "issued_at": "2026-04-11T00:00:00Z",
  "expires_at": "2026-04-12T00:00:00Z",
  "max_uses": 10,
  "scope": {
    "roles": ["gpu-worker"],
    "labels": {"pool": "inference-a100"}
  },
  "signature": "ed25519:<base64>"
}
```

Tokens are generated by the enrollment authority and distributed through secure channels (secrets managers, CI pipelines, operator scripts).

### Node Attestations

A node attestation is a signed document proving fleet membership.

```json
{
  "kind": "attestation/v1",
  "subject": "fp:node-fingerprint",
  "subject_public_key": "ed25519:<base64>",
  "issued_by": "fp:registry-fingerprint",
  "issued_at": "2026-04-11T08:00:00Z",
  "expires_at": "2026-07-11T08:00:00Z",
  "roles": ["gpu-worker"],
  "labels": {"pool": "inference-a100"},
  "signature": "ed25519:<base64>"
}
```

Attestations have an expiry. Nodes must re-enroll or renew before expiry. This bounds the damage window from a compromised node key.

### Peer Verification

When node A connects to node B:

1. node A presents its identity document and attestation;
2. node B verifies the attestation signature against the enrollment authority's public key;
3. node B checks that the attestation has not expired;
4. node B checks the revocation list;
5. if all checks pass, node B trusts node A for the scoped roles in the attestation.

No direct identity exchange between peers. Trust flows through the enrollment authority.

## Design: Key Rotation

### Node Key Rotation

A node can rotate its signing key without losing fleet membership.

Rotation flow:

1. node generates a new signing key;
2. node signs a key rotation request with the old key, including the new public key;
3. node submits the rotation request to the enrollment authority;
4. enrollment authority verifies the old key signature;
5. enrollment authority issues a new attestation for the new key;
6. enrollment authority records the rotation in its ledger;
7. node persists the new key and attestation;
8. node re-signs all local items with the new key.

The old key is retained in a `retired_keys/` directory for verification of historical signatures. It is never used for new signing.

### Enrollment Authority Key Rotation

The enrollment authority can rotate its own signing key using a chain-of-trust model.

1. the old key signs a key rotation document containing the new public key;
2. all fleet nodes receive the rotation document;
3. fleet nodes verify the rotation document against the old key;
4. fleet nodes add the new key to their trusted set;
5. after a grace period, the old key is retired.

This is similar to how DNSSEC key rollovers work.

## Design: Revocation

### Node Revocation

The enrollment authority maintains a revocation list.

```json
{
  "kind": "revocation_list/v1",
  "issued_by": "fp:registry-fingerprint",
  "issued_at": "2026-04-11T08:00:00Z",
  "revoked": [
    {
      "subject": "fp:compromised-node-fingerprint",
      "revoked_at": "2026-04-10T12:00:00Z",
      "reason": "key_compromise"
    }
  ],
  "signature": "ed25519:<base64>"
}
```

Fleet nodes periodically fetch the revocation list from the enrollment authority. The fetch interval determines the maximum window between revocation and enforcement.

For immediate revocation, the enrollment authority can push revocation events to connected nodes.

## Design: Ephemeral Workers

Ephemeral GPU workers (spot instances, autoscaling pools) need a streamlined enrollment path.

### Pre-Authorized Pools

The enrollment authority can pre-authorize a pool with a multi-use join token scoped to specific roles and labels. Worker launch scripts use the token during `init`:

```
ryeosd init --join https://registry.example.com --token-file /run/secrets/pool-token
ryeosd serve
```

### Automatic Deregistration

When an ephemeral worker shuts down gracefully, it notifies the enrollment authority to deregister. If it crashes, the enrollment authority detects the absence through heartbeat timeout and marks the node as inactive.

Inactive nodes are not revoked. They are simply removed from the active enrollment ledger. If the same node identity reappears, it can re-enroll without a new token if its attestation has not expired.

## Relationship To Existing Architecture

### Capability Refs

Cluster bootstrap and capability refs (from the v2 future-evolution doc) are complementary. Cluster bootstrap establishes node-level trust. Capability refs establish execution-level authorization. A node that is trusted for fleet membership may still require a capability ref to execute a specific tool on behalf of a specific agent.

### CAS And Registry

The enrollment authority can be co-located with the registry. The registry already stores signed objects and has an auth layer. Adding enrollment and attestation documents to the registry is a natural extension.

### Node Vault

Join tokens and attestation documents can be stored in the node vault for secure persistence. The vault already handles encrypted secret storage at the node level.

Cluster bootstrap is also a prerequisite for [Node Vault V3 — Cluster Vault](node-vault.md#v3--cluster-vault-future). Vault federation (secret replication policies, centralized management, satellite distribution) depends on attestation-based trust between nodes. Without fleet enrollment, there is no trust basis for replicating secrets across nodes.

## Evaluation Criteria

Before building any of this:

1. Is manual identity exchange actually painful at current scale? If not, defer.
2. Are there more than a handful of nodes that need to trust each other? If not, defer.
3. Are ephemeral workers a real deployment pattern today? If not, defer.
4. Can the current file-based bootstrap be extended to cover the use case without the full enrollment protocol? If yes, extend rather than redesign.

The file-based trust exchange in the v3 daemon bootstrap is designed to be sufficient for the near term. This document preserves the design for when scale demands it.
