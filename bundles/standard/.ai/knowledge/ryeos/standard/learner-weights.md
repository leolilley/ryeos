# Durable learner weights

Learned parameters are durable model artifacts, not secrets. They can be much
larger than configuration values and must participate in content-addressed
retention, synchronization, and lineage. Do not place weights in the runtime
vault. The vault is for bounded secret/configuration values and opening its
sealed key/value envelope is intentionally not the data path for model blobs.

## Storage contract

| Concern | Primitive | Why |
|---|---|---|
| Immutable checkpoint bytes | bundle-event CAS attachment | Content-addressed, deduplicated, and retained by the event's object closure. |
| Latest checkpoint | head event of the model's chain | One signed, atomic authority; no second mutable “latest” pointer can disagree. |
| Lineage and metrics | bundle-event payload | The same append commits the parent digest, metrics, and checkpoint attachment. |
| Runtime file | explicit project-relative path | The project chooses its model layout; RyeOS does not invent a hidden runtime directory. Project snapshots include normal project paths. |
| UI/debug summary | thread artifact | Optional metadata only; never the checkpoint store. |

An event and all of its attachment descriptors are canonical CAS data. Each
descriptor contains a name, blob hash, byte length, and optional media type.
The object-closure walker follows attachment blob hashes, so terminal-history
retention, export, remote synchronization, and GC treat the event and its
checkpoint as one durable graph.

Historical bundle events without attachments remain readable with their exact
canonical bytes and hashes. This is historical-data integrity, not a second
active persistence implementation.

## Project paths

The runtime supplies a project-relative `source_path` when appending and a
project-relative `destination_path` when restoring. These are normal,
project-owned paths such as:

```text
models/arc_actor_v1/checkpoint.json
```

RyeOS does not create `.ai/runtime`, write generated bytes into the bundle, or
select a path for the project. Paths are confined below the execution's pinned
project root, every component is opened without following symlinks, and writes
are atomic. Normal project snapshot rules therefore capture a materialized
checkpoint when the project wants it included in its source/result state.

## Manifest capability

The bundle declares append and scan authority for its checkpoint event kind:

```yaml
runtime_authority:
  bundle_events:
    - event_kind: learner_weights_event
      operations: [append, scan]
```

No runtime-vault capability is needed for learner weights.

## Persist a checkpoint

First write the checkpoint to the project path chosen by the project. Then
append one event with the file as an attachment:

```json
{
  "event_kind": "learner_weights_event",
  "chain_id": "arc_actor_v1",
  "event_type": "weights_updated",
  "expected_chain_head_hash": "<previous event hash or null>",
  "idempotency_key": "<stable update id>",
  "payload": {
    "parent_checkpoint_hash": "<previous attachment blob hash or null>",
    "format": "json",
    "metrics": {"games_seen": 110, "success_rate": 0.37}
  },
  "attachments": [
    {
      "name": "checkpoint",
      "source_path": "models/arc_actor_v1/checkpoint.json",
      "media_type": "application/json"
    }
  ]
}
```

The callback reads the exact regular file without following symlinks, bounds
it to 32 MiB, stores it in CAS, and advances the event-chain head while holding
the daemon mutation guard. A crash before the signed head advance leaves only
an unreferenced blob that normal GC can remove; a successful append makes the
blob reachable immediately. `expected_chain_head_hash` provides optimistic
concurrency and the idempotency key makes a retry return the original event.

An append response includes the event and attachment descriptor. Persist the
returned `event_hash` or discover the newest event by reading the chain head;
the attachment `blob_hash` is the canonical checkpoint digest.

Limits per append:

- 16 attachments;
- 32 MiB per attachment;
- 128 MiB total attachment input;
- existing bundle-event payload and canonical object limits still apply.

## Load the latest checkpoint

Read the model chain newest-first. If it is empty, initialize the project's
default weights. Otherwise materialize the `checkpoint` attachment from the
head event:

```json
{
  "event_kind": "learner_weights_event",
  "event_hash": "<head event hash>",
  "attachment_name": "checkpoint",
  "destination_path": "models/arc_actor_v1/checkpoint.json",
  "replace": true
}
```

Use the `runtime.bundle_events_materialize_attachment` callback. The daemon
revalidates the event object and blob hash, verifies the event belongs to the
callback's effective bundle and authorized event kind, and atomically writes
the chosen destination without following symlinks.

## Lifecycle

1. Read the newest event in the model's bundle-event chain.
2. Materialize its checkpoint attachment into the project-owned model path, or
   initialize defaults when the chain is empty.
3. Update the model in memory and write the next checkpoint file.
4. Append one event with metrics, parent checkpoint hash, and the checkpoint
   attachment, using the observed head as `expected_chain_head_hash`.
5. Optionally publish a small thread artifact containing only the event hash,
   checkpoint hash, and metrics.

This leaves one source of truth: the signed event-chain head selects the
latest immutable checkpoint, and that event's CAS closure owns the bytes.
